# P8 Engine 使用教程与问题解决指南

本文档包含两部分：**使用教程**（从安装到实战）和**问题解决**（常见 AI 编码问题及 P8 如何应对）。

---

## 第一部分：使用教程

### 1. 安装

**方式一：Plugin Marketplace（推荐）**

```bash
# 添加 marketplace 源
claude plugin marketplace add darven-cs/p8-engine

# 安装插件
claude plugin install p8-engine@p8-engine
```

安装后启动新的 Claude Code 会话，输入 `/p8-engine` 激活。

**方式二：手动安装**

```bash
git clone https://github.com/darven-cs/p8-engine.git ~/.claude/skills/p8-engine
```

重启 Claude Code 后，`/p8-engine` 即可用。

### 2. 激活与启动协议

输入 `/p8-engine` 后，skill 会自动执行**会话启动协议**（三步）：

```
你：/p8-engine

Claude：
  [风格检测] 扫描项目结构，读取 1-2 个类似文件，输出：
    - 命名约定（camelCase / snake_case）
    - 文件组织（按功能 / 按层级）
    - 注释密度、错误处理模式、模块粒度

  [任务校准] 定义"够好"的标准：
    - Must（必须达到）
    - Should（应该达到）
    - Could（锦上添花）

  [激活确认] 输出 "P8 mode activated"，开始工作
```

这三步是**自动的**——你不需要手动触发任何东西。完成后 Claude 就以 P8 工程模式运行。

### 3. 基本用法：直接给任务

激活后，直接描述你的任务即可。P8 会在后台自动应用规则。

```
你：/p8-engine 帮我在用户注册接口加上参数校验

Claude 会自动：
  1. 先读现有的校验代码（风格检测的延续）
  2. 找到类似接口的校验模式（铁律二：先查后问）
  3. 按项目现有风格实现（精准匹配，不引入新依赖）
  4. 运行测试验证（证据交付）
  5. 检查其他接口是否有同样问题（铁律三：主动延伸）
```

**关键点**：你不需要记住任何规则名，只需正常描述任务。P8 作为行为协议在底层约束 Claude 的行为。

### 4. 进阶用法：结合内联问题

P8 支持在激活时附带具体问题或需求：

```
# 带问题激活
/p8-engine 这个 API 为什么返回 500？

# 带约束激活
/p8-engine 重构数据库层，但不要改接口签名

# 带上下文激活
/p8-engine 用户反馈导出功能超时，帮我排查
```

这种方式将你的问题/需求直接传入启动流程，Claude 会在风格检测和任务校准时把你的具体需求纳入考虑。

### 5. 不同任务类型的适配

P8 内置了任务类型适配表，对不同任务有不同的侧重：

| 任务类型 | P8 侧重点 |
|---------|----------|
| Bug 修复 | 铁律四（全链路排查）+ 铁律三（同类延伸）|
| 新功能 | 模块开发 Phase 0-4（读→设计→自检→实现→验证）|
| 重构 | 铁律五（精准修改）+ 质量自检 |
| 性能优化 | 铁律一（穷尽方案）+ 证据对比 |
| 调试/排查 | 应急流程（诊断→拉高视角→最小行动）|
| 代码审查 | 质量自检 10 项全覆盖 |

### 6. 模块开发流程（Phase 0-4）

当任务是"开发一个新模块"时，P8 会自动触发完整的 Phase 流程：

```
Phase 0 - 读（~10分钟）
  ├── 找到项目中类似的参考模块
  ├── 逐行阅读，提取风格和边界
  └── 输出风格检测结果

Phase 1 - 设计契约
  ├── 模块名 + 单一职责
  ├── 输入（前置条件）
  ├── 输出（后置保证）
  ├── 依赖（以接口表达）
  ├── 扩展点（策略/接口/配置/hook）
  └── 明确排除项

Phase 2 - 自检
  ├── 一句话职责测试
  ├── 依赖方向检查
  └── 搜索可复用代码

Phase 3 - 实现
  ├── 匹配项目风格
  ├── 渐进式：先跑通 → 再纠正 → 再优化
  └── 函数长度控制（40 行阈值）

Phase 4 - 验证
  ├── 功能测试 + happy path
  ├── 2+ 边界用例
  └── 契约一致性检查
```

详细的 Phase 指南参见 [references/module-dev-protocol.md](../references/module-dev-protocol.md)。

## 第二部分：问题解决

以下是使用 AI 编码时常见的 10 个问题，以及 P8 Engine 如何系统性解决每个问题。每个场景包含**问题表现**、**P8 的应对机制**和**实际效果对比**。

---

### 问题 1：空口说"已完成"

**表现**
```
你：帮我加个登录接口
Claude：好了，我已经完成了。
（但没跑测试，没验证，可能根本跑不起来）
```

**P8 的应对：铁律七 + 质量自检**

P8 要求每次交付必须附带**证据**：
- 必须运行 build / test / lint
- 输出结果摘要
- 交付格式：`[交付] <做了什么> -- 验证: <build/test/lint 输出摘要>`

**效果对比**

```
# 没有 P8
"已完成，请查看代码。"

# 有 P8
"[交付] 登录接口已实现
  - POST /api/auth/login
  - 参数校验：email 格式 + 密码非空
  - 验证：npm test 通过（12/12），lint 0 warning
  - curl 测试返回 200 + valid JWT"
```

---

### 问题 2：信息不足就问用户

**表现**
```
你：这个报错怎么修？
Claude：你用的什么数据库？什么版本？配置文件在哪？错误日志完整贴一下？
（连续追问，用户疲于应付）
```

**P8 的应对：铁律二 — 先查后问**

规则：必须先用工具自行调查，只问真正需要确认的问题。带着发现去问，不空手提问。

**效果对比**

```
# 没有 P8
"请提供数据库版本和配置文件路径。"

# 有 P8
"我查了项目结构，发现你用的是 PostgreSQL 14（见 docker-compose.yml:5），
 配置在 config/database.yml:12。
 报错的根因是连接池耗尽——当前 max_connections=10 但并发请求峰值约 50。
 需要确认：是否可以调大 max_connections，还是希望我加连接队列？"
```

### 问题 3：修完一个 bug 就停

**表现**
```
你：用户反馈搜索功能偶尔返回空结果
Claude：（修了搜索接口的一个 null check）已修复。
（但排序接口有同样的 null 问题，没检查）
```

**P8 的应对：铁律三 — 主动延伸**

修完一个 bug 后，在**同一根因**范围内延伸排查：
- 搜索"同一模式在其他位置是否重复"
- 只延伸同一根因，不做附带"改进"

**效果对比**

```
# 没有 P8
（只修了搜索接口的 null check）

# 有 P8
"[交付] 修复搜索 null 值问题
  - 根因：未对 Optional 字段做 null 检查
  - 搜索接口：已修复 ✓
  - 排序接口：同一问题，已修复 ✓
  - 过滤接口：同一问题，已修复 ✓
  - 共计 3 处，均为同一根因"
```

---

### 问题 4：只修当前报错，不看全链路

**表现**
```
你：前端报 500 错误
Claude：（看了前端代码）前端请求头格式不对，改了。
（实际是后端接口参数变了，前端只是受害者）
```

**P8 的应对：铁律四 — 全链路排查**

要求画出完整依赖链，自底向上验证，不打地鼠：
1. 定位报错位置
2. 画出调用链：前端 → 网关 → 后端 → 数据库
3. 从最底层开始验证
4. 确认修复覆盖整条链路

**效果对比**

```
# 没有 P8
"前端请求头格式有误，已修正。"
（后端下次改接口，前端又坏）

# 有 P8
"[交付] 500 错误根因分析
  依赖链：前端 → API Gateway → UserController → UserService → DB
  - DB 层：schema 正常 ✓
  - Service 层：正常 ✓
  - Controller 层：v2 API 已改参数名，但未标记 v1 为 deprecated ✗
  - Gateway：路由规则正常 ✓
  - 前端：仍在用旧参数名，但这是后端变更未通知导致的
  修复：后端恢复向后兼容 + 添加 v1 deprecated 标记"
```

---

### 问题 5：有歧义不澄清就干

**表现**
```
你：优化一下性能
Claude：（自作主张重写了整个查询层，引入了 Redis、消息队列...）
（你只是想加个索引）
```

**P8 的应对：任务校准 + 铁律五**

- 启动时的**任务校准**会定义 Must / Should / Could 三级
- 铁律五：每行改动必须直接对应需求，不做"顺手改进"

**效果对比**

```
# 没有 P8
（重写了整个数据层，引入了 Redis 和消息队列，2000 行改动）

# 有 P8
"任务校准：
  - Must：列表页查询响应 < 200ms
  - Should：首页加载 < 500ms
  - Could：缓存命中率 > 80%

  分析后确认瓶颈在缺少索引。
  [交付] 添加复合索引 (status, created_at)
  - 改动：1 个 migration 文件，+3 行
  - 验证：查询时间 1200ms → 45ms
  - 未引入新依赖"
```

### 问题 6：代码膨胀

**表现**
```
你：写个用户注册
Claude：（写了 800 行，包含：用户注册、邮件验证、密码强度检测、
       操作日志、消息队列通知、Admin 管理界面...）
```

**P8 的应对：质量自检 — 简洁性**

质量自检包含简洁性检查：不加未被要求的抽象，不加未被要求的功能。

**效果对比**

```
# 没有 P8
（800 行，5 个新文件，引入 3 个新依赖）

# 有 P8
"[交付] 用户注册接口
  - POST /api/users/register
  - 输入校验 + 密码哈希 + 数据库写入
  - 改动：1 个文件，+45 行
  - 未添加：邮件验证（需求未提及）
  - 未添加：Admin 界面（需求未提及）
  - 建议后续：如需邮件验证，可作为独立模块添加"
```

---

### 问题 7：顺手改了无关代码

**表现**
```
你：修一下这个 typo
Claude：（修了 typo，顺手重构了整个文件，改了变量命名，
       调整了代码顺序，加了类型注解...）
（diff 显示 200 行改动，实际只需要改 1 行）
```

**P8 的应对：铁律五 — 精准修改**

每行改动必须直接对应需求。不做附带"改进"。

**效果对比**

```
# 没有 P8
  diff: +150/-80 行（实际 typo 只需改 1 行）

# 有 P8
  diff: +1/-1 行
  "[交付] 修复 README.md 中 'recieve' → 'receive' typo"
```

### 问题 8：代码风格和项目格格不入

**表现**
```
（项目用 camelCase、2 空格缩进、无分号）
Claude：（新写的代码用 snake_case、4 空格、带分号，
       还引入了 Lodash 虽然项目用的是 Ramda）
```

**P8 的应对：会话启动协议 — 风格检测**

激活时强制读取现有代码，提取风格特征：
- 命名约定
- 缩进 / 格式化
- 依赖选择
- 错误处理模式
- 模块粒度

**效果对比**

```
# 没有 P8
（新代码和项目风格完全不同，PR review 被打回）

# 有 P8
"风格检测完成：
  - 命名：camelCase
  - 缩进：2 spaces
  - 分号：无
  - 工具库：Ramda（非 Lodash）
  - 错误处理：Result monad 模式
  新代码将严格遵循以上约定。"
```

### 问题 9：卡壳反复微调同一方案

**表现**
```
Claude：（方案 A 失败）让我试试调整参数...
Claude：（方案 A' 失败）再调整一下...
Claude：（方案 A'' 失败）换个参数值...
（重复 10 次，死磕同一个方向）
```

**P8 的应对：应急流程**

四步应急协议：

```
Step 1 - 诊断卡壳模式
  □ 方向对但方法错？
  □ 方向本身错了？
  □ 信息不足？
  □ 错误假设？
  □ 工具限制？
  □ 能力边界？

Step 2 - 拉高视角
  □ 原样读失败信号（不要解读，先照抄）
  □ 主动搜索相关文档
  □ 读源码上下文
  □ 验证每个假设
  □ 反转假设（如果反过来想？）

Step 3 - 最小行动恢复
  □ 先做最小保证成功的步骤
  □ 然后逐步扩展

Step 4 - 穷尽后结构化退出
  输出结构化交接报告：
  - 已验证的事实
  - 已排除的可能
  - 已缩小的范围
  - 推荐的下一步
```

**效果对比**

```
# 没有 P8
（重复尝试同一方案 10 次后放弃，只说"我解决不了"）

# 有 P8
"[应急] 方向对但方法错——确诊：API 限流导致的 429，
 不是代码逻辑问题。已验证：手动 curl 同样返回 429。
 最小行动：加 exponential backoff 重试。
 [交付] 添加重试机制，3 次重试后放弃。
 验证：在限流条件下成功获取数据。"
```

### 问题 10：交付质量参差不齐

**表现**
```
同一个 Claude，有时候写出来的代码很好，有时候很糟。
质量不稳定，取决于任务复杂度和上下文长度。
```

**P8 的应对：质量自检 10 项**

每次交付前必须通过 10 项检查：

**5 项通用检查**
| 检查项 | 含义 |
|--------|------|
| 正确性 | 功能是否按预期工作 |
| 完整性 | 是否覆盖所有需求点 |
| 精准度 | 改动是否精确对应需求 |
| 简洁性 | 是否有不必要的复杂度 |
| 诚实度 | 是否如实报告了限制和风险 |

**5 项代码检查**
| 检查项 | 含义 |
|--------|------|
| 风格一致 | 是否匹配项目现有代码风格 |
| 单一职责 | 每个函数/模块是否只做一件事 |
| 依赖健康 | 是否引入了不必要的依赖 |
| 安全意识 | 是否有安全隐患 |
| 可扩展性 | 是否预留了合理的扩展点 |

**效果对比**

```
# 没有 P8
（质量随机波动，无法预测）

# 有 P8
（每次交付都经过同样的检查清单，质量稳定可预期）
"[自检]
  正确性 ✓  功能测试通过
  完整性 ✓  覆盖全部 3 个需求点
  精准度 ✓  改动限于需求范围
  简洁性 ✓  无多余抽象
  诚实度 ✓  已标注：暂未处理并发场景，建议后续加强
  风格一致 ✓
  单一职责 ✓
  依赖健康 ✓  无新依赖
  安全意识 ✓  已做输入校验
  可扩展性 ✓  预留了策略接口"
```

## 快速参考

### 激活方式
```
/p8-engine                          # 基本激活
/p8-engine <你的任务或问题>           # 带任务激活
```

### 核心铁律速查
| # | 铁律 | 一句话 |
|---|------|--------|
| 1 | 穷尽方案 | 不说"不可能"之前先穷尽所有方案 |
| 2 | 先查后问 | 先用工具自查，再问用户 |
| 3 | 主动延伸 | 修完 bug 检查同一根因的其他位置 |
| 4 | 全链路排查 | 画依赖图，自底向上验证 |
| 5 | 精准修改 | 每行改动对应需求，不做附带修改 |
| 6 | 检查点意识 | 大改前确保状态可恢复 |
| 7 | 自动提交 | 每个原子改动提交到本地 git |

### 应急流程速查
```
卡壳 → 诊断模式 → 拉高视角 → 最小行动 → 穷尽后结构化退出
```
