# 模块开发协议 — 从需求到交付的完整路径

这是 `SKILL.md` 中"模块开发协议"章节的详细展开。主文件给你摘要，这里给你肌肉。

---

## 为什么需要协议？

没有协议的代码开发路径通常是：
1. 看了一眼需求 → 直接开始写
2. 写到一半发现接口不对 → 推倒重来
3. 写完后发现和项目现有风格不一致 → 再改
4. 交付后发现边界 case 没覆盖 → 再改
5. 一周后想扩展 → 发现当初没预留扩展点 → 又推倒重来

**协议的价值：** 把"返工"从执行后移到执行前。5 分钟设计 > 30 分钟返工。

---

## Phase 0：理解现有代码（强制 10 分钟）

### 为什么不能跳过？

你不是在建新房子，你是在装修已有的房子。不了解这个房子就动锤子 = 打穿承重墙。

### 执行步骤

**1. 找到参照物（2-3 分钟）**

找到项目中和你要做的模块最相似的现有模块：
- 同类型功能（也是 CRUD？也是计算引擎？也是事件处理器？）
- 同一层（也在 Service 层？Controller 层？）

**2. 精读参照物（5-8 分钟）**

不是扫一眼，是逐行理解：
- 文件结构怎么组织的？
- 类/函数的命名遵循什么规律？
- 错误怎么处理和返回的？
- 依赖怎么注入的？
- 注释的密度和风格？

**3. 输出风格检测结果**

```
[风格检测] 基于 <参照文件路径>
命名约定：<具体示例，如 getUserById → camelCase，类名 PascalCase>
错误处理：<throw Error / return {ok, error} / Promise.reject>
依赖方式：<构造器注入 / 函数参数 / 模块导入>
分层规范：<Service 层不能直接操作 DB，通过 Repository>
注释习惯：<JSDoc / 行内注释 / 无注释>
测试约定：<__tests__ 目录 / *.test.ts 文件 / 无>
```

## Phase 1：设计契约

### 接口优先于实现

在写任何实现代码之前，先定义这个模块的"契约"——它承诺对外提供什么，需要外部提供什么。

**这不是 Java 的强制接口语法要求，这是强迫你在动手前想清楚。**

### 模块设计模板

```
[模块设计] <模块名>

一句话职责：
  这个模块负责 <做什么>，不负责 <不做什么>

输入（前置条件）：
  - 参数名：类型，约束（如：不能为空，长度 1-100，正整数）
  - 调用时机：<什么情况下调用这个模块>

输出（后置保证）：
  - 成功时：返回类型和结构
  - 失败时：抛出什么异常 / 返回什么错误结构

依赖（需要外部提供什么能力）：
  - <依赖名>：<用接口/类型描述，不是具体实现>
  - 示例：需要一个能"通过 ID 查用户"的能力，不关心是 MySQL 还是 Redis

扩展点（未来可能变化的地方）：
  - <变化点 1>：预留方式（策略模式 / 接口 / 配置 / Hook）
  - 示例："定价规则可能变" → 用 PricingStrategy 接口预留

不做（明确排除）：
  - 这个模块不负责 <X>（X 属于 <另一个模块>）
```

### 如何识别扩展点？

问以下三个问题：

1. **"如果需求变了..."**：产品最可能要求我改什么？定价规则、通知渠道、存储后端
2. **"如果环境变了..."**：从测试环境到生产环境，什么会不一样？外部 API、配置参数
3. **"如果规模变了..."**：用户量 10x，什么地方会成为瓶颈？数据库查询、缓存策略

## Phase 2：软件工程自检

在写实现之前，对着 Phase 1 的设计过一遍：

### 职责检查

- 一句话描述通过了吗？（不含"还有"和"以及"）
- 如果不通过，把这个模块一分为二

### 依赖方向检查

画出依赖图（哪个模块依赖哪个）：
```
你的模块 → [依赖 A] → [依赖 B]
```
- 依赖是指向稳定方向（接口/抽象）还是不稳定方向（具体实现）？
- 如果模块 A 依赖模块 B，当 B 改变时 A 需要改变吗？

### 已有功能检查

在实现之前搜索项目：
- 有没有现成的工具函数/服务可以复用？
- 有没有我要实现的逻辑已经在别处实现过了？

**搜索关键词：** 功能的核心动词（validate, calculate, format, parse...）

## Phase 3：实现原则

### 用主人的风格写，不是你认为"更好"的风格

你的审美在这里不重要。**项目一致性 > 你认为的最佳实践。**

例外：如果现有风格有安全隐患，先实现用主人的风格，然后在 [战果] 中提出来，让主人决定。

### 渐进式实现

1. 先让它跑起来（最小可用版本）
2. 再让它跑对（覆盖所有约定的行为）
3. 最后让它跑好（性能、边界、安全）

不要同时做三件事。先跑起来才能验证方向对不对。

### 函数长度控制

单个函数超过 40 行时停下来问：
- 有没有可以提取的子步骤？
- 有没有重复的模式？
- 这个函数是不是做了超过一件事？

### 提前返回（Guard Clause）

```python
# 不好：嵌套地狱
def process(user):
    if user is not None:
        if user.is_active:
            if user.has_permission:
                do_work(user)

# 好：Guard Clause
def process(user):
    if user is None: return
    if not user.is_active: return
    if not user.has_permission: return
    do_work(user)
```

## Phase 4：验证清单

### 功能验证

- [ ] 走了 Happy Path（正常输入，预期输出）
- [ ] 走了边界 Case（空值、最大值、并发、重复调用）
- [ ] 走了失败 Case（非法输入、依赖失败、网络超时）

### 接口一致性验证

- [ ] 输出格式和 Phase 1 设计一致（类型、结构）
- [ ] 错误格式和 Phase 1 设计一致（异常类型、错误码）
- [ ] 调用约定和参照物一致（调用方式、参数顺序）

### 质量罗盘（代码版）

- [ ] 职责清晰：一句话描述通过？
- [ ] 风格一致：和参照物风格匹配？
- [ ] 可扩展：扩展点预留了吗？
- [ ] 依赖健康：依赖方向正确？
- [ ] 安全：输入验证、权限检查到位？

## Phase 5：模块复盘（必须写）

完成一个模块后，写入 `builder-journal.md`：

```
[模块复盘] <YYYY-MM-DD> <模块名>
职责：<一句话，和 Phase 1 的一句话描述一致>
架构决策：<选择了什么方案，为什么不选其他的>
扩展点：<留了什么接口，未来怎么扩展>
已知局限：<这个设计在什么条件下会撑不住>
下次参考：<写类似模块时，哪个决定值得重用>
```

### 好的复盘 vs 坏的复盘

**坏的（过程描述）：**
> 先试了 A 方案，发现不行，然后改成 B 方案，B 方案中间遇到了 X 问题，最后解决了。

**好的（结论沉淀）：**
> 选择 Repository 模式隔离数据层，因为这个模块以后可能从 MySQL 迁移到 MongoDB。已知局限：当前 Repository 接口不支持复杂查询，复杂聚合查询需要额外设计。

## 常见陷阱速查

### 陷阱 1：接口设计时考虑了实现

```
# 坏：接口泄露了 SQL 细节
def get_users(sql_where_clause: str) -> List[User]

# 好：接口描述意图
def get_active_users_by_region(region: str) -> List[User]
```

### 陷阱 2：过早优化

常见症状：还没让模块跑起来，就开始考虑缓存/并发/性能。

**顺序：跑起来 → 跑对 → 跑快。** 在跑起来之前，性能优化是浪费时间。

### 陷阱 3：依赖了具体实现

```python
# 坏：业务逻辑直接依赖 MySQL
class OrderService:
    def __init__(self):
        self.db = MySQLConnection(host="localhost")

# 好：依赖接口
class OrderService:
    def __init__(self, order_repo: OrderRepository):
        self.order_repo = order_repo
```

### 陷阱 4：错误信息不够用

当模块失败时，调用方需要知道：
- **发生了什么**（错误类型）
- **为什么发生**（原因）
- **可以怎么办**（建议操作，如果有）

只抛 `Exception("error")` = 给调用方一张白纸 = 调试地狱。

### 陷阱 5：忘记了并发场景

如果这个模块可能被并发调用：
- 有没有共享状态？（类变量、全局变量）
- 有没有非原子操作（读-改-写）？
- 有没有外部资源竞争（文件锁、数据库行锁）？

## 模块开发 checklist（打印版）

```
开工前：
  [ ] 读了参照物（1-2 个现有类似模块）
  [ ] 输出了[风格检测]
  [ ] 读了 user-style.md
  [ ] 写了[模块设计]（一句话职责 + 输入/输出/依赖/扩展点/不做）

实现前：
  [ ] 软件工程自检通过（SOLID 5 问）
  [ ] 搜索了是否有现成实现可复用

实现中：
  [ ] 用主人的风格写，不是我认为好的风格
  [ ] 渐进式：先跑起来，再跑对，再跑快

完成后：
  [ ] Happy Path 验证通过
  [ ] 边界 Case 覆盖
  [ ] 接口和 Phase 1 设计一致
  [ ] 质量罗盘 10 问通过
  [ ] 写了[模块复盘]到 builder-journal.md
  [ ] 如果发现主人新风格偏好，更新 user-style.md
```
