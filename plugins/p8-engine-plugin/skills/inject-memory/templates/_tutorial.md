# AI 记忆系统搭建教程（可套用到任何项目）

> 这是一套让 Claude / AI 协作者在多次会话间保持"项目记忆"的轻量方案。本目录（`memory/`）本身就是一个活生生的样例，可以直接整份复制到新项目。

---

## 一、这套系统解决什么问题

AI 协作者（Claude / Cursor 等）的单次会话上下文是短暂的。下次开新对话它就忘了：
- 这个模块为什么这么设计
- 这个项目有哪些"踩过的坑"，不要再踩
- 某块代码之前的演进历史、遗留问题
- 公开接口、数据流、依赖关系

通常的补救是放一个大 `CLAUDE.md` / `AGENTS.md` 往里堆信息，很快就臃肿到不可读，且 AI 扫描成本变高。

**方案**：用一个结构化的 `memory/` 目录，配合轻量 `_index.md` 索引 + 每次加载索引的硬约束，让 AI 能"按需深入"。索引扫一眼知道**有什么、在哪、当前状态**，需要细节时再展开读。

---

## 二、核心设计原则

| # | 原则 | 理由 |
|---|---|---|
| 1 | **索引轻量、详情下沉** | 索引每次会话都加载，必须能在一屏内扫完；详情只在需要时展开 |
| 2 | **按"查询意图"拆文件，不按"信息类型"堆文件** | 你找东西时问的是"这模块架构是什么"、"这模块怎么接入的"，不是"按时间列出所有修改" |
| 3 | **Why + How to apply 比 What 重要** | What 代码里自己有；AI 需要的是背景原因和未来判断依据 |
| 4 | **不重复 git log 能给的信息** | 开发时间线详表、谁改了什么行——这类查 git 即可，记忆里只留提炼后的结论 |
| 5 | **结构随规模升级，不要一开始过度设计** | 模块小就单文件；内容过半屏再考虑拆子目录 |

## 三、物理目录结构

```
memory/
├── _index.md            # 索引（始终加载，保持轻量）
├── _tutorial.md         # 本教程（可选保留）
├── modules/             # 模块记忆
│   ├── small_module.md              # 小模块/参考类：单文件
│   └── big_module/                  # 大模块：子目录
│       ├── overview.md
│       ├── design.md
│       ├── integration.md
│       └── known_issues.md
├── bugs/                # 跨模块通用踩坑（一坑一文件）
│   └── {tech}_{symptom}.md
└── progress/            # 会话级进度（可选）
    ├── current-status.md
    └── session-YYYY-MM-DD.md
```

## 四、每类文件记什么

### 4.1 `modules/` — 模块记忆

模块是"一块有明确边界的业务/技术功能"，不等于一个类/一个文件。

#### 拆分粒度判断

| 情况 | 用什么结构 |
|---|---|
| 单文件讲完不超半屏 | `modules/{type}_{name}.md` 单文件 |
| 内容需要分节、多人会从不同角度问起 | `modules/{name}/` 子目录 + 4 份文件 |

**命名类型前缀**（单文件时建议带）：
- `project_` — 项目内核心业务模块（如 `project_fishing_pool.md`）
- `ref_` — 外部依赖或共享基础设施使用方式（如 `ref_static_define.md`）
- `feedback_` — 用户反馈/团队约定的强规则

#### 大模块的 4 份文件各回答什么

| 文件 | 回答什么 | 不要放什么 |
|---|---|---|
| `overview.md` | 这个模块有什么类/接口/数据？新人读 5 分钟能上手 | 详细设计原因、踩坑细节 |
| `design.md` | 为什么这样做？改动某条之前该注意什么？ | 接口签名、接入方式 |
| `integration.md` | 和项目其他部分怎么对接？新人复用接入路径要抄哪几个点？ | 模块内部实现细节 |
| `known_issues.md` | 还有哪些未修的坑，修的建议方向 | 已修的 bug（去 git log / `bugs/` 找） |

**原则**：如果某一份内容不足半屏，合并进 `overview.md`，不要造空文件。

### 4.2 `bugs/` — 跨模块通用踩坑

**进入标准**：同时满足以下两条。
1. **跨模块可能再次踩到**（语言特性、环境限制、框架 bug、第三方库陷阱）
2. **从代码里看不出来为什么要这么写**（规避规则隐藏在原因里）

**不进入**：只发生在某个模块一次，下次不会复现的——记在 `modules/{模块}/known_issues.md` 或 git commit 说明里就够了。

**命名**：`{tech}_{symptom}.md`，例如：
- `php_or_operator_precedence.md`
- `redis_ttl_vs_business_expire.md`
- `react_stale_closure_in_useeffect.md`
- `docker_layer_cache_invalidation.md`

### 4.3 `progress/` — 会话进度（可选）

| 文件 | 用途 |
|---|---|
| `current-status.md` | 当前在做什么、卡在哪、下一步计划——会话中断后恢复用 |
| `session-YYYY-MM-DD.md` | 每次会话结束时追加：今天改了什么、为什么、遗留什么 |

如果项目是你一个人做、节奏紧凑，可以不启用；多人协作或 AI 经常被中断则很有用。

## 五、`_index.md` 怎么写

**约束**：200 行内，能在一屏半内扫完。表格优先，段落次之。

**必含五块**：

```markdown
# 项目记忆索引

> 本索引始终加载。先扫这里，再按条目跳详细文件。

## 一、模块记忆
[表格：模块名 | 路径 | 状态 | 最后更新 | 一句话]

## 二、踩过的坑
[表格：坑 | 路径 | 触发条件 | 一句话规则]

## 三、当前进度
[指向 progress/current-status.md；或"目录暂空"]

## 使用规则
[5 条左右：什么时候读、什么时候更新索引]

## 模块记忆文件结构约定
[大模块 4 份文件各回答什么]
```

索引**不要写**：
- 实际的内容（那是详情文件的事）
- 变更历史（在对应详情文件的"变更历史"节里）
- 长篇段落（一行 150 字以内的一句话 hook 就够）

## 六、在协作工具里硬约束 AI 去使用

光建目录没用，必须让 AI 每次都加载索引。在 `CLAUDE.md` / `AGENTS.md` / `.cursorrules` 等项目指令里明确写：

```markdown
## 记忆系统使用规范

### 记忆文件位置
- 总索引：`memory/_index.md`（始终加载）
- 详细记忆存放目录：`/memory/`

### 你必须遵守的记忆规则
1. 开始任务前，先扫描 `_index.md`，确认涉及模块的当前状态和已知约束
2. 任何对模块的变更（新增功能、修改接口、改变依赖），任务完成后必须：
   - 在对应模块的记忆文件中追加一条变更历史（日期 + 简要说明）
   - 如果公开接口变化，更新接口表
   - 如果模块状态、最后更新日期改变，同步更新 `_index.md`
3. 当用户说"记一下"，立即执行上述更新
4. 新增模块时，自动创建模块记忆文件并在 `_index.md` 添加索引
5. 新发现通用踩坑时，按"跨模块 + 代码看不出原因"标准判断是否进 `bugs/`
```

没有这段硬约束，AI 知道有这套系统也不会主动用。

## 七、从零搭建的 7 步

### 步骤 1 — 建目录骨架

```bash
mkdir -p memory/{modules,bugs,progress}
touch memory/_index.md
```

### 步骤 2 — 在项目主指令文件里加使用规范

`CLAUDE.md` / `AGENTS.md` 里贴入上一节的硬约束段落。

### 步骤 3 — 写初始 `_index.md` 骨架

可直接复制本教程第十一节"索引模板"。

### 步骤 4 — 识别第一批要记录的模块

不要试图一次记录所有模块。识别标准：
- 最近 1-2 个月有大改的
- AI 最可能被问到的
- 有非显而易见设计决策的
- 有踩过明显坑的

每个候选模块先建单文件 `modules/project_{name}.md`，内容不多就一直单文件。

### 步骤 5 — 第一个模块达到半屏以上时，升级为子目录

```bash
mkdir memory/modules/big_module
# 按 overview / design / integration / known_issues 拆内容
rm memory/modules/project_big_module.md  # 拆完清理原文件
```

### 步骤 6 — 抽离通用踩坑到 `bugs/`

每个模块文件里"遇到的问题"章节内，按"跨模块 + 代码看不出原因"标准筛选。满足的抽到 `bugs/{tech}_{symptom}.md`，模块文件里保留链接或删除。

### 步骤 7 — 同步更新 `_index.md`

每新增一个 modules 文件 / bugs 文件 / 升级子目录，都在索引的对应表里加一行。

## 八、日常维护节奏

| 时机 | 动作 |
|---|---|
| 每次任务开始前 | AI 自动读 `_index.md`（由 CLAUDE.md 约束） |
| 修改了某模块的公开接口 | 更新对应 `overview.md` 的接口表 + `_index.md` "最后更新"列 |
| 做了一个"有原因的设计选择" | 在 `design.md` 追加一条决策表 |
| 踩到跨模块通用的坑 | 新建 `bugs/{tech}_{symptom}.md` + `_index.md` 坑表加一行 |
| 修复了某个 bug | 不记——git log 已经有了，除非想提炼通用教训 |
| 用户说"记一下" | 立即执行上述动作，这是强信号 |
| 用户说"忘了" | 找到对应文件删除条目或更新状态 |
| 模块下线/废弃 | 在对应 `overview.md` 顶部标 "**已废弃**"，保留文件做历史参考；或整体移至 `memory/_archive/` |

## 九、反面教材（别这么做）

| 做法 | 为什么不行 |
|---|---|
| 把整个模块文档塞进 `_index.md` | 索引每次加载，臃肿后 AI 扫描成本爆炸 |
| 按"开发阶段 1 / 2 / 3"详细记录时间线 | 下次查记忆不会按时间线查；git log 已经有 |
| 把 "WHAT 代码做了什么" 写进记忆 | 代码自己能说；记忆只记 WHY 和 How to apply |
| 把每个 bug 都独立一个文件 | 只有跨模块通用的才进 `bugs/`；一次性 bug 进 git commit 信息 |
| 一开始就给每个模块建子目录 | 绝大多数模块永远不够半屏，空文件更难维护 |
| 记录团队成员名字和 RACI | 这是项目管理工具的事，不是 AI 记忆的事 |
| 记录"今天的心情"或废话 | 每行都要能被未来查询复用 |

## 十、几个判断准则

**单文件还是子目录？**
> 单文件超过半屏（~60 行）且内容能分出明显主题 → 子目录。否则单文件。

**进 `bugs/` 还是 `known_issues`？**
> 跨模块可能复现 + 代码看不出原因 → `bugs/`。只在这个模块 + 有注释说明 → `known_issues`。

**值不值得记？**
> 问自己："下次开新会话时，没有这条记忆会让 AI 做错什么？" 答不上来 → 不记。

**要不要放索引？**
> 问自己："一眼扫索引能判断出某任务涉不涉及这个模块 / 这个坑 吗？" 不能 → 详情下沉，不要进索引。

**旧记忆还准不准？**
> 推荐前验证：记忆说"X 函数在某文件"——grep 一下还在不在。记忆是时间点的快照，代码才是真相。

## 十二、套用到新项目的最快路径

```bash
# 1. 复制本 memory/ 目录骨架到新项目
cp -r /path/to/this/memory /path/to/new-project/
cd /path/to/new-project/memory

# 2. 清空项目特定内容，保留骨架
rm -rf modules/* bugs/* progress/*
# _index.md 清空表格行但保留结构
# _tutorial.md 保留原样

# 3. 在新项目的 CLAUDE.md / AGENTS.md 加入"记忆系统使用规范"段落

# 4. 开始为第一批模块建单文件记忆
```

从这一刻起，每次让 AI 干活它都会先读索引、按约定更新——记忆系统就活起来了。
